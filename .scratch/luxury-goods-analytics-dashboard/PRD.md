# 二奢数据分析仪表盘（Next.js + Prisma + Neon）

## Problem Statement

当前仓库已经具备基于 Neon PostgreSQL 的竞品已售分析能力，并有一套 Python / Streamlit 与一个轻量 Next.js 前端雏形，但尚未形成面向二奢收货定价场景的完整仪表盘。业务需要围绕“套利 / 定价”模型，消费由爬虫持续写入云端数据库的价格数据，输出可直接辅助收货决策的市场均价、建议收货价、短期风险预警与趋势图，而不是停留在通用查询或单一价格走势页。

团队希望以轻量、可部署、低维护的方式交付这个能力：复用现有仓库与数据表，前端使用 Next.js App Router，数据库访问统一到 Prisma，部署直接通过 Vercel CLI 执行，而不是引入额外后台模板或冗余中台层。

## Solution

在现有仓库 `web/` 应用中新增一个面向二奢定价的 Next.js 仪表盘：前端提供型号与成色输入、核心指标卡片、风险诊断和 30 天趋势图；后端通过 Server Action 调用 Prisma，在 Neon 中聚合最近 30 天价格数据，并按成色折损系数、目标安全边际与近 3 日跌幅规则计算建议收货价与风险等级。

整体方案以“一个可直接演示的端到端垂直切片”为原则：从数据库连接、分析算法、页面交互、图表展示到构建与部署都打通。部署路径使用 Vercel CLI，环境变量在 Vercel 项目中配置 `DATABASE_URL`，数据库连接统一使用 Neon pooler 地址，避免 serverless 连接数与冷启动问题。

## User Stories

1. 作为二奢收货人员，我希望输入一个型号编号后立即看到市场分析结果，以便快速判断是否值得继续谈价。
2. 作为二奢收货人员，我希望按成色选择“全新 / 95新 / 90新”，以便系统根据商品状态调整建议收货价。
3. 作为二奢收货人员，我希望看到最近市场均价，以便把收货谈判锚定在真实市场水平上。
4. 作为二奢收货人员，我希望系统自动给出建议最高收货价，以便减少人工口算和经验误差。
5. 作为二奢收货人员，我希望看到风险等级，以便在价格快速下行时主动压低收货价。
6. 作为二奢收货人员，我希望看到风险原因说明，以便理解系统为何把某个型号判为“预警”或“高危”。
7. 作为二奢收货人员，我希望看到最近 30 天价格走势，以便识别短期波动和异常下跌。
8. 作为二奢收货人员，我希望在没有该型号数据时收到明确提示，以便及时改用别的型号关键词。
9. 作为分析人员，我希望系统聚合的是最近 30 天有效样本，以便分析窗口稳定且可复现。
10. 作为分析人员，我希望算法显式使用成色折损系数，以便后续能独立调参。
11. 作为分析人员，我希望算法显式使用目标安全边际，以便把套利空间写成清晰规则。
12. 作为分析人员，我希望近 3 日跌幅超过阈值时自动触发风险收紧，以便在市场转弱时降低库存风险。
13. 作为分析人员，我希望风险规则与建议收货价计算共享同一套输入数据，以便避免两个结果口径不一致。
14. 作为开发者，我希望数据库访问统一通过 Prisma，以便和 Next.js Server Action 的运行模式一致。
15. 作为开发者，我希望 Prisma Client 在 serverless 环境中以 singleton 方式复用，以便避免开发环境热重载下重复建连。
16. 作为开发者，我希望数据库连接字符串使用 Neon pooler 地址，以便降低 Vercel serverless 连接抖动。
17. 作为开发者，我希望分析逻辑封装为单一可测试接口，以便后续扩展成更多风险因子时不必重写页面。
18. 作为开发者，我希望页面组件与分析逻辑解耦，以便将来替换图表库或增加更多视图时影响面可控。
19. 作为开发者，我希望输入校验在服务端也执行，以便避免空 `modelId` 或非法参数直接进入查询。
20. 作为开发者，我希望原始 SQL 查询通过 Prisma 的安全参数化能力执行，以便兼顾灵活性与安全性。
21. 作为开发者，我希望 TypeScript 类型能明确描述分析结果结构，以便前后端共享同一数据契约。
22. 作为维护者，我希望本地能跑通 `npm run build`，以便在上线前先验证类型与构建完整性。
23. 作为维护者，我希望部署通过 Vercel CLI 完成，以便把部署动作纳入终端工作流与脚本化流程。
24. 作为维护者，我希望在 Vercel 中显式注入 `DATABASE_URL`，以便运行环境与本地行为一致。
25. 作为维护者，我希望部署的是仓库当前分支而不是手工复制文件，以便发布过程可追溯。
26. 作为产品负责人，我希望页面只展示必要分析信息，以便首屏聚焦在收货定价而非做成泛 BI 平台。
27. 作为产品负责人，我希望不引入厚重 admin 模板，以便界面保持轻量、加载快、维护简单。
28. 作为未来开发者，我希望风险阈值与成色系数具备集中定义点，以便后续从硬编码升级为配置化。
29. 作为未来开发者，我希望能在当前仪表盘基础上继续扩展更多品类或品牌视角，而不推翻现有接口。
30. 作为未来开发者，我希望复用现有 `web/` 应用而不是另起一个前端工程，以便减少仓库分裂。

## Implementation Decisions

- 实施位置放在现有 `web/` Next.js 应用内，不新建第二个前端项目。
- 页面采用 Next.js App Router + TypeScript，前端视图聚焦单页仪表盘，不扩展为多模块后台。
- UI 采用 Tailwind CSS + Tremor UI；若当前 `web/` 还未装配 Tailwind，需要先补齐基础样式管线，再接 Tremor 组件。
- 数据访问从当前 `@neondatabase/serverless` 直连查询模式切换或收敛到 Prisma，以便统一数据库抽象与 serverless 生命周期管理。
- 新建一个深模块“市场分析服务”，它接收 `modelId` 与 `condition`，返回统一的 `MarketAnalysis` 结构，负责封装数据查询、均价计算、收货价建议和风险评估。
- 新建一个深模块“风险评估规则集”，负责封装成色折损系数、基础安全边际、近 3 日跌幅阈值与风险等级映射，避免这些业务常量散落在页面组件中。
- 新建一个深模块“价格时间序列查询器”，负责在 Neon 中读取最近 30 天、按日聚合后的原始价格时间序列，供分析服务消费。
- 分析服务内部可继续使用 Prisma 的 `$queryRaw`，但所有动态参数必须走参数化插值，不能拼接裸 SQL 字符串。
- 默认市场窗口为最近 30 天；后续若扩展自定义时间窗，优先在分析服务扩参，而不是复制一套新 action。
- 当前 PRD 将 `condition` 规范为 `全新`、`95新`、`90新` 三档；服务端要对超出枚举值的输入做兜底或显式报错。
- 建议收货价算法先采用固定规则：市场均价 × 成色系数 × 安全边际；这是首版可演示模型，不追求一次性覆盖所有业务变量。
- 风险规则首版以“最近 3 个观测点中，最新价格相较第 3 个点跌幅超过 5%”作为预警门槛；若数据点不足，则回退为默认安全状态。
- 风险等级先保留 `安全`、`预警`、`高危` 三档，但首版至少实现 `安全` 与 `预警` 的可验证行为；`高危` 可先作为保留枚举，待后续补充更强规则。
- 趋势图展示使用 Tremor 图表组件，输入数据直接消费 `chartData`，不在客户端再次做聚合。
- 页面交互以客户端组件承载，用户提交后调用 Server Action；避免在客户端直接暴露数据库访问逻辑。
- 错误体验以可理解的业务提示为主，例如“未找到该型号基准数据”或“缺少数据库连接”，避免把底层异常直接抛给用户。
- Prisma Client 采用 singleton 模式，兼容 Vercel serverless 与本地开发热更新。
- `DATABASE_URL` 必须使用 Neon pooler 连接串，不接受普通直连地址作为默认推荐。
- 构建验证以 `npm run build` 为发布前硬门槛；未通过构建不得继续部署。
- 部署流程使用 Vercel CLI 执行，包括项目关联、环境变量注入、预览部署与正式部署；不把“登录 Vercel 控制台手工点 Deploy”作为默认路径。
- 部署前代码推送到 Git 分支仍然保留，但发布动作以 Vercel CLI 为准，保证命令行可重复执行。

## Testing Decisions

- 好的测试应只验证外部行为与稳定契约，不绑定具体 SQL 语句排列、组件内部 state 细节或图表库内部实现。
- 优先测试的模块是“市场分析服务”，因为它承载核心业务价值，且最适合用固定输入输出做断言。
- “风险评估规则集”应有独立单测，覆盖正常市场、近 3 日跌幅超过 5%、样本不足、非法成色输入等场景。
- “价格时间序列查询器”应通过数据库夹具或 mock Prisma 响应验证聚合结果映射，不测 Prisma 自身行为。
- 页面层测试重点放在用户可见行为：输入型号、选择成色、触发分析、展示指标、展示错误提示。
- 若当前仓库中缺少前端测试基建，首轮至少保证 TypeScript 类型检查与 `npm run build` 通过；随后再补 Vitest / React Testing Library。
- 可参考仓库现有 Python 侧测试风格：优先验证纯逻辑模块与稳定边界条件，而不是过度依赖端到端浏览器测试。
- 部署前验证至少包含：依赖安装成功、Prisma Client 生成成功、构建通过、关键环境变量存在。

## Out of Scope

- 不在本 PRD 内实现完整后台管理系统、用户权限系统或多角色登录。
- 不在本 PRD 内实现爬虫本身、抓取调度或新增数据源接入。
- 不在本 PRD 内重做现有竞品导入链路、快照归档规则或 Python 导入脚本。
- 不在本 PRD 内实现复杂机器学习定价模型；首版仅实现可解释规则引擎。
- 不在本 PRD 内实现多页面导航、品牌库管理、供应商管理或库存管理。
- 不在本 PRD 内实现移动端原生适配或 App 打包。
- 不在本 PRD 内把所有旧前端查询页统一迁移到 Prisma；仅处理本次仪表盘涉及部分。
