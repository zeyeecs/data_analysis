# 跨渠道对比分析（sjkx）

## Problem Statement

运营 **f店**、**r店** 两家 Shopify **在售库存** 与 Vestiaire **已售记录**，需要基于历史 **快照** 做 **对比分析**：首要为 **店铺间对照**（两店 **目录聚合对照**），其次为以 V 为参照的 **Vestiaire 参照对照**。数据来自飞书 xlsx，飞书会定期清理文件且清理时机不确定；**渠道内 ID** 不能跨店 join，也尚无 **商品对齐**。团队需要可重复的对照逻辑（而非每次手写 SQL），并保留飞书已删但曾导入的 **快照** 以供跨月分析。当前导入已支持 **数据归档** 与 **按日覆盖**，但 **最近邻配对**、**目录聚合指标** 与 **可复用对照查询** 尚未落地；**汇率换算** 亦未实现。

## Solution

在 Neon PostgreSQL 中维护 F / R / V 三张业务表的 **数据归档**，提供仓库内 **可复用对照查询**（SQL 视图或参数化脚本），封装 **最近邻配对**（±3 天）、**目录聚合指标**（件数、均价、最低/最高、中位数、标价总额）及 **店铺间对照** / **Vestiaire 参照对照** 两种分析路径。导入支持 **分表导入** 与 **导入自检**；分析阶段默认 f店/r店 **挂牌价**、V **成交参考价**，跨币种时先按 `currency` 分组，待 **汇率换算**（CNY + 快照日公开中间价）实现后再统一为 **基准货币**。远期 API/页面应复用同一套对照语义，本阶段不交付接口。

## User Stories

1. 作为运营人员，我希望从飞书 F/R/V 目录导入 xlsx 到数据库，以便集中查询三家渠道数据。
2. 作为运营人员，我希望多次导入时保留不同 `snapshot_date` 的历史 **快照**，以便飞书删文件后仍能分析上月数据。
3. 作为运营人员，我希望对同一 `snapshot_date` 再次导入时覆盖该日数据而非整表清空，以便修正单日导出错误。
4. 作为运营人员，我希望只导入 f店或 r店或 V 其中一张表，以便灵活更新单侧数据。
5. 作为运营人员，我在只更新 f店未更新 r店时仍能导入成功，以便接受中间状态并在对照前 **导入自检**。
6. 作为运营人员，我希望推荐 f店与 r店同一批次导入，以便 **店铺间对照** 两侧快照尽量新鲜。
7. 作为运营人员，我希望导入在下载失败时不删除该日已有数据，以免误删 **快照**。
8. 作为运营人员，我希望文件名无法解析 `snapshot_date` 时收到警告，以便知晓该批数据归入 NULL 日期桶。
9. 作为分析师，我希望明确 `item_id` 仅为 **渠道内 ID**，以便不误做跨店逐件 join。
10. 作为分析师，我希望在无 **商品对齐** 时仍能做 **目录聚合对照**，以便按品牌/型号/成色比较两店。
11. 作为分析师，我希望对 f店某一 `snapshot_date` 自动在 r店找到 **最近邻配对**（≤3 天），以便两店快照时间不完全一致时仍能 **店铺间对照**。
12. 作为分析师，当 r店与 f店最近快照相差超过 3 天时，我希望系统不生成无效配对，以便避免错配时间窗。
13. 作为分析师，我希望对 **目录聚合对照** 默认看到件数、均价、最低价、最高价、中位数与标价总额，以便全面比较价量。
14. 作为分析师，我希望 f店/r店对照使用 **挂牌价**，以便与在售标价语义一致。
15. 作为分析师，我希望 V 侧 **Vestiaire 参照对照** 使用 **成交参考价** 的 `price` 字段，以便与挂牌口径区分清晰。
16. 作为分析师，我希望 V 侧不按 `sold_at` 另设时间窗，以便与 F/R 一样在 **对照快照对** 的 `snapshot_date` 上聚合。
17. 作为分析师，我希望以 V 的 `snapshot_date` 为锚，分别在 f店、r店各找 **最近邻配对** 做 **Vestiaire 参照对照**，以便两店分别相对 Vestiaire 参照。
18. 作为分析师，我希望在 V 存在多 `currency` 时先按币种分组查看，以便在 **汇率换算** 未就绪时仍能分析。
19. 作为分析师，我希望将来按 **快照** 日公开中间价将 V 价格换算为 **基准货币** CNY，以便跨币种与 f店/r店 **挂牌价** 可比。
20. 作为分析师，我希望通过参数（如锚定 `snapshot_date`、对照类型）执行 **可复用对照查询**，以便不必每次重写配对逻辑。
21. 作为分析师，我希望在 Neon 中对手写 SQL 探索仍保持开放，以便在标准查询外做临时分析。
22. 作为维护者，我希望 `--reset` 仍能清空并重建 F/R/V 表，以便灾难恢复或 schema 变更后全量重导。
23. 作为维护者，我希望可选删除遗留 `feishu_files` 元数据表，以便与现行「仅业务行数据」模型一致。
24. 作为未来开发者，我希望对照规则集中在可复用模块/SQL 中，以便 API/页面上线时不重复实现 **最近邻配对** 规则。
25. 作为运营人员，我希望 README 说明默认 **累积保留** 与 **按日覆盖** 行为，以便团队正确执行日常导入。

## Implementation Decisions

### 模块划分（深模块优先）

| 模块 | 职责 | 变更类型 |
|------|------|----------|
| **飞书表导入器** | 列举飞书 xlsx、下载、解析、**按日覆盖** 写入 F/R/V；支持分目录与 `--reset` | 已 largely 实现；将 `delete_snapshot_date` 等逻辑收口为可测函数 |
| **快照日解析器** | 从文件名解析 `snapshot_date`（含多种中文/数字日期格式） | 提取自现有 xlsx 解析；接口稳定、单测友好 |
| **最近邻配对器** | 给定锚定表与日期、目标表可用日期列表，返回日历差最小且 ≤3 天的配对日；无配对时返回空 | **新建**；纯函数，不依赖数据库，供 SQL 与远期 API 共用 |
| **目录聚合构建器** | 按 `brand` / `model` / `condition`（及 V 的 `currency` 可选）生成聚合 SELECT：件数、均价、min/max、中位数、`sum(price)` | **新建**；以 SQL 片段或 CTE 生成器实现；F/R 用 **挂牌价**，V 用 **成交参考价** |
| **对照查询目录** | 参数化 **店铺间对照**（F 锚定→R 配对）、**Vestiaire 参照对照**（V 锚定→F、V 锚定→R 各一条配对链） | **新建** SQL 脚本或 `VIEW` + 文档化参数；消费 **最近邻配对器** 的语义（可在 SQL 中用窗口函数实现同等逻辑，须与 Python 模块结果一致） |
| **汇率仓储（远期）** | 按 `snapshot_date` + 币种存储公开中间价，换算为 **基准货币** CNY | **新建**；遵循 ADR-0002；数据提供商与缺日回填规则实现前可占位 schema |
| **遗留同步路径** | 旧 `feishu_files` 全量同步 | 不扩展；文档标明 deprecated，与 PRD 范围无关除非维护者显式清理 |

### 架构与行为

- **数据归档（累积保留）** 与 **快照覆盖（按日覆盖）** 以 ADR-0001 为准；导入器不得在默认路径 `TRUNCATE` 整表。
- **最近邻配对** 上限 3 天（含边界需在设计中明确：差 3 天是否可配对——领域对话视为可配对，实现时 `abs(delta) <= 3`）。
- **店铺间对照**：默认 F 为锚、R 为配对侧（与术语示例一致）；允许查询参数交换锚定侧但语义须文档化。
- **Vestiaire 参照对照**：V 锚定；对 F、R 各独立 **最近邻配对**，互不替代。
- **目录聚合指标** 全集：count、avg、min、max、median、sum(price)；PostgreSQL 用 `percentile_cont(0.5)` 实现中位数。
- **汇率换算** 本 PRD 不实现具体爬取/API，仅预留表结构与对照查询中的可选 CNY 列扩展点；未提供汇率时查询退化为按 `currency` 分组。
- **分析交付（现阶段）** 以 SQL 为主；不在本 PRD 实现 HTTP API 或 UI（**分析交付（规划）**）。
- 同一 `snapshot_date` 同日多文件：后导入文件覆盖先导入该日全部行（导入顺序依赖飞书列表顺序）。

### Schema（远期汇率）

- 新增汇率表：至少 `(rate_date DATE, currency TEXT, cny_rate NUMERIC)` 唯一约束 `(rate_date, currency)`；`rate_date` 对齐 **快照** 的 `snapshot_date`。
- 不在 F/R 表增加 `currency` 列（当前 **挂牌价** 已为 CNY 口径）。

### 配对逻辑（与原型等价的状态机）

锚定侧日期 `anchor_d`，候选日期列表 `candidates`：

```
paired = argmin |d - anchor_d|  for d in candidates
if paired is None or |paired - anchor_d| > 3: 无配对
else: 对照快照对 = (anchor_d, paired)
```

平手（相同最小差）时取较早日期或较晚日期——实现须固定一种规则并写入测试。

## Testing Decisions

- **原则**：只测可观察行为（配对结果、解析出的日期、聚合行数与指标值），不测 SQL 字符串拼装细节或飞书 HTTP  mock 除非必要。
- **建议单测模块**：**快照日解析器**、**最近邻配对器**（含 >3 天无配对、±3 边界、平手规则、空候选列表）。
- **目录聚合构建器**：可用内存 SQLite/PostgreSQL fixture 插入少量行，断言聚合结果；或 golden SQL 对比固定 fixture 的输出。
- **飞书表导入器**：集成测试成本高，本阶段以手工导入验证为主；若测则仅测 **按日覆盖** 的删除+插入行为（可用 mock cursor）。
- **对照查询目录**：至少一条 **店铺间对照**、一条 **Vestiaire 参照对照** 的端到端 SQL 用例（fixture 数据），验证配对日与聚合列存在。
- **先验**：仓库尚无 `tests/`；新建 `tests/` 采用 pytest，与 `requirements.txt` 一致。

**模块测试意愿（待维护者确认）**：默认可为 **快照日解析器** 与 **最近邻配对器** 编写单测；导入器与 SQL 端到端用例可选。

## Out of Scope

- **商品对齐** 与跨平台逐 `item_id` / SKU join。
- Web API、前端页面（仅预留语义一致的对照模块）。
- **汇率换算** 的具体数据源接入、节假日缺日回填策略（ADR-0002 保留决策，实现另开）。
- 外部 BI 工具配置、Metabase 仪表盘。
- 扩展 `seller_price`、`buyer_fee` 等 V 字段的对照口径。
- 飞书文件自动删除检测与库内 **快照** 主动清理策略。
- 实时/sync 旧版 `feishu_files` 管道的新功能。
